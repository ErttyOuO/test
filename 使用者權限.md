## 使用者權限（admin / sponsor / agent / user）— 現況與串接指引

本專案目前沒有後端；所有登入/註冊/權限/使用者資料皆以 `localStorage` 模擬。
因此「權限」主要是靠 `auth.js` 裡的 **role 欄位** 來判斷。

### 角色一覽

| role | 說明 | 可進入的頁面範圍 |
|------|------|-----------------|
| `user` | 一般使用者 | 前台功能（訂閱、錢包、簽到、AI 諮詢、貼圖商城…） |
| `sponsor` | 贊助商 | 前台 + 贊助商後台（sponsor-dashboard） |
| `agent` | 保險員 | 前台 + 保險員後台（未來擴充） |
| `admin` | 管理員 | 全部頁面 |

### Demo 帳號（auth.js → `ensureDemoAccounts()`）

| 帳號 | 密碼 | role | 說明 |
|------|------|------|------|
| `superuser` | `0000` | user | 一般使用者 |
| `sponsorA` | `1111` | sponsor | 贊助商 Demo |
| `agentA` | `2222` | agent | 保險員 Demo |
| `admin` | `9999` | admin | 管理員 |

### 服務入口頁（portal.html）

`portal.html` 是所有角色共用的「首頁入口」，登入後依據 `getCurrentUserRole()` 自動過濾並顯示該角色可進入的頁面卡片。不同角色看到的分類不同：

- `user`：只看到「一般功能」
- `sponsor`：一般功能 + 贊助商後台
- `agent`：一般功能 + 保險員後台
- `admin`：全部（一般功能 + 贊助商後台 + 保險員後台 + 管理員後台）

若要新增頁面，只需在 `portal.html` 的 `ALL_PAGES` 陣列加一筆，指定 `roles` 即可。

### 1) 權限判斷的核心：`localUsers[].role`

**權限來源不是 `authUser`、也不是 `user_data`**，而是存在 `localStorage.localUsers`（使用者清單）每筆使用者物件上的：

- `role: 'user' | 'sponsor' | 'agent' | 'admin'`

對應程式位置：

- 註冊新使用者時，預設 `role = 'user'`
	- `auth.js` → `registerLocalUser(displayName, username, password)`
- Demo 帳號的 role 在 `auth.js` → `ensureDemoAccounts()` 中指定

角色查詢與權限閘道：

- `getUserRole(userId)`：從 `localUsers` 找 userId 的 role（找不到就回 `'user'`）
	- 特例：`local-admin` 永遠回 `'admin'`（防止測試時被誤改）
- `getCurrentUserRole()`：取目前登入者的 role
- `requireRole(...allowedRoles)`：頁面進入點的權限檢查
	- 若未登入：回傳 false（由登入 Gate 顯示登入畫面）
	- 若 role 不符合：顯示「權限不足」遮罩
	- 特例：若 role 是 `sponsor` 或 `agent` 但進入了不屬於自己的後台 → 自動導回 `portal.html`

> 結論：你要調整「某使用者是 admin / sponsor / agent / 普通人」→ 改 `localUsers` 裡該使用者的 `role`。

---

### 2) `user_data` 與 `role` 的差異（常見混淆點）

本專案同時存在兩套資料：

1. **帳號清單（localUsers）**：
	 - Key：`localUsers`（陣列）
	 - 內容：`{ id, displayName, username, password, role, createdAt, lastLoginAt, ... }`
	 - 用途：登入/註冊與「角色權限」判斷（`role`）

2. **每位使用者的資料（user_data_<userId>）**：
	 - Key：`user_data_<userId>`（物件）
	 - 用途：coins、VIP、簽到、廣告每日上限、以及一些廣告相關旗標（例如 `is_sponsor`）

其中：

- `role`：用於進入後台頁面的權限判斷（最重要）
- `user_data.is_sponsor`：偏「資料/狀態旗標」，常用於顯示與廣告 SDK 的補充邏輯

> 結論：想讓使用者能進 sponsor 後台 → 一定要設 `role='sponsor'`。
> `is_sponsor=true` 建議一起設，但它不是進入頁面的主要 gate。

---

### 3) 目前各頁面如何做權限限制（入口點）

（以下都是在 `DOMContentLoaded` 時呼叫 `requireRole(...)`）

- 服務入口頁：所有角色皆可進入（依 role 顯示不同卡片）
	- `portal.html` → 無 `requireRole`，僅檢查 `isLoggedIn()`

- 管理員後台：只允許 `admin`
	- `admin-dashboard.html` → `requireRole('admin')`
	- `admin-ads.html` → `requireRole('admin')`
	- `sticker-admin.html` → 目前未加 requireRole，建議補上 `requireRole('admin')`

- 贊助商後台：允許 `sponsor` 或 `admin`
	- `sponsor-dashboard.html` → `requireRole('sponsor','admin')`

- 保險員後台：允許 `agent` 或 `admin`（待開發）
	- `agent-dashboard.html` → 建議用 `requireRole('agent','admin')`

- 個人頁面 Sponsor 徽章顯示：看 `getCurrentUserRole()` 是否為 `sponsor`
	- `profile.html`

---

### 4) 贊助商資料（Sponsor Profile / Wallet）在哪裡

贊助商相關資料由 `ad-sdk.js` 統一管理，存放在：

- `sponsor_profiles`：`{ [userId]: { user_id, company_name, verified, frozen, created_at } }`
- `sponsor_wallets`：`{ [userId]: { balance, total_deposited, total_spent } }`
- `sponsor_wallet_history_<userId>`：交易紀錄（陣列）

目前初始化點：

- `auth.js`：在 Demo 帳號 `sponsorA` 建立時，會自動補 sponsor profile + wallet
- `sponsor-dashboard.html`：載入時 `ensureSponsorProfile()` / `ensureSponsorWallet()`
	- 同時會把 `user_data.is_sponsor = true`

### 5) 未來要做「註冊平台」：如何直接給相應權限（建議做法）

#### A. 你要寫入的最小必要欄位

只要你能在「註冊 / 登入成功後」確保下列資料正確，就可以直接套用現有的 `requireRole(...)`：

1. `localUsers` 中該使用者的 `role`
	 - `admin` / `sponsor` / `agent` / `user`
2. `authUser`（目前登入者資訊）
	 - 由 `setAuthenticatedUser(user)` 寫入 `{ id, displayName, username }`

#### B. Sponsor 角色建議同時補齊的資料（避免 UI/資料不同步）

若註冊平台決定某人是 `sponsor`，建議同時完成：

- `localUsers[].role = 'sponsor'`
- `user_data_<id>.is_sponsor = true`
- `sponsor_profiles[id]`（至少含 verified/frozen/created_at）
- `sponsor_wallets[id]`

這樣就算使用者尚未進入 `sponsor-dashboard.html`，管理員後台匯總/顯示也會一致。

#### C. 建議的「資料來源」模式（避免前端被隨意改角色）

若未來有真正後端：

- 由後端決定 role（例如註冊後預設 user；審核通過後升 sponsor；少數帳號設 admin）
- 前端只負責接收 role 並寫入 localUsers（Demo 模式）

> 注意：目前是純前端 localStorage Demo，任何人打開 DevTools 都能改 `localUsers.role`。
> 上線環境必須把權限判斷搬到後端（例如 API 以 token role/claims 判斷）。

---

### 6) 快速排查：覺得權限怪怪時先看哪裡

1. `localStorage.localUsers`：該使用者的 `role` 是否正確
2. `localStorage.authUser`：目前登入者 id 是否正確
3. `localStorage.user_data_<id>`：是否有被不同帳號的資料覆蓋（id 對不上）
4. 若 sponsor/agent 被導去 portal.html：表示你進入了不屬於自己的後台頁面（`requireRole` 會自動導回入口頁）

---

### 7) 新增頁面到入口頁的步驟

1. 在 `portal.html` 的 `ALL_PAGES` 陣列中加一筆：
   ```js
   {
       id: 'agent-dashboard',
       title: '保險員後台',
       desc: '管理客戶、保單、業績報表',
       icon: '🏥',
       href: 'agent-dashboard.html',
       roles: ['agent', 'admin'],     // 哪些角色可以看到這張卡片
       category: 'agent',              // 分類 key
       badge: '保險員'
   }
   ```
2. 在該頁面的 `DOMContentLoaded` 加上 `if (!requireRole('agent','admin')) return;`
3. 完成！portal.html 會自動依 role 顯示/隱藏
