# AI 語音情緒陪伴助手（AI 保險員）專案復盤：從 MVP 到多模態架構定案

## 0. 版本演進總覽

* **V1** ：定時截圖 + 本機 DeepFace 情緒分類 + Gemini 文字回覆（基礎 MVP）。
* **V2** ：修正本機機器學習環境衝突與 Gemini API 模型名稱迭代問題，確保 MVP 穩定可跑。
* **V3** ：嘗試調教 DeepFace，加入各種平滑演算與極端分數門檻策略，試圖解決嚴重誤判問題。
* **V4** ：放棄本機情緒模型，架構大重構，將多模態判讀（視覺+語意）統一交由 Gemini 處理。
* **V5** ：加入語音輸入（STT）、手機內網測試支援，並確立「AI 保險員」的具體應用情境。
* **V6** ：加入 PDF 保單上傳解析、可拖曳浮動鏡頭、持久化歷史對話與整體 UX 優化，並嘗試導入串流回應。

---

## 1. 專案起點：一開始想做什麼

這個專案最初的構想，是建立一個 **「能根據使用者當下心情，動態調整回覆語氣」的 AI 聊天助手** 。

當時的具體需求與第一版 MVP 目標包含：

* **視覺情緒偵測** ：透過電腦鏡頭畫面，判斷使用者是開心、不開心還是未知狀態。
* **情緒影響對話** ：將偵測到的情緒與使用者對話一起送給 LLM，讓 AI 能針對「開心」給予活潑延伸、「不開心」給予溫和同理的支持。
* **前後端分離與彈性部署** ：要求前端為純 HTML/CSS/Vanilla JS，並且具備「API Base URL 切換」功能。這讓前端可以輕易部署在 GitHub Pages 上，而後端無論是在 localhost 還是未來的雲端伺服器，前端都能一鍵切換對接。
* **技術選型** ：一開始選擇了 FastAPI 作為後端，並指定使用 **DeepFace** 進行本機端的情緒視覺辨識，再串接 Gemini Python SDK 產生文字回覆。

這個起點很明確：試圖將「本機影像辨識」與「雲端文本生成」結合，提供一個具備基礎同理心的網頁 Demo。

## 2. 第一版是怎麼做出來的

最初版本的骨架，是圍繞著「定時截圖分析」與「雙模式部署」建立的。

* **前端實作** ：使用 Web API (`navigator.mediaDevices.getUserMedia`) 取得鏡頭畫面。設定一個計時器，每隔 3 秒利用 Canvas 將 Video 畫面畫下來，轉成 JPEG Base64/Blob，透過 `fetch` POST 給後端。同時實作了 `localStorage` 來記憶 API Base URL。
* **後端實作** ：FastAPI 提供 `/health`、`/config` 與 `/analyze` 三個端點。接收到圖片後，不落地儲存，直接在記憶體中交給 DeepFace 解析 `dominant_emotion`。
* **初步資料流** ：前端每 3 秒送圖 -> 後端 DeepFace 進行二分類 -> 加入最近 3 次的滑動平均避免跳動 -> 組合情緒狀態與 Prompt 呼叫 Gemini -> 回傳前端顯示。

## 3. 開發過程中的主要階段 (Phase / Evolution)

專案在開發過程中經歷了典型的 Vibe Coding 演進，遇到瓶頸就快速轉向。以下是幾個關鍵的演進階段與因果關係：

### Phase 1：基礎 MVP 建立與本地模型跑通

* **目標** ：讓前後端對接，確認攝影機畫面能傳到後端並獲得 AI 回應。
* **動作** ：建立 FastAPI 路由、完成 CORS 設定、實作前端的 Start/Stop 控制項。
* **因果轉折** ：初步跑通後，立刻迎來了本地端 ML 環境的「依賴地獄」，迫使我們進入 Phase 2 的環境除錯。

### Phase 2：處理環境衝突與 API 版本迭代

* **目標** ：解決套件報錯與 API 失效問題。
* **動作** ：在後端處理 `numpy.float32` 的 JSON 序列化報錯；並透過腳本動態抓取可用模型，將失效的 Gemini 模型名稱更新至 `gemini-2.5-flash`。
* **因果轉折** ：環境雖然穩定了，但實際測試發現 DeepFace 準確度極低，導致 AI 回覆邏輯錯亂，因此進入 Phase 3 的模型調教期。

### Phase 3：深陷情緒辨識泥淖與調參

* **目標** ：解決使用者明明在笑，卻一直被判定為 `unhappy` 的問題。
* **動作** ：改用 `retinaface` 提升人臉抓取率；放棄 `dominant_emotion` 改寫極端分數門檻（例如 `happy >= 5%` 即判定開心）；甚至將誤判的 `surprise` 強制視為 `happy`。
* **因果轉折** ：因為 DeepFace 在真實場景下誤判嚴重，且過度消耗本機資源，後續所有調參其實都只是延後重構的徒勞。這促使我們做出重大決策，徹底放棄本地視覺模型，進入 Phase 4。

### Phase 4：斷尾求生與架構大重構（The Pivot）

* **目標** ：捨棄無效技術債，擁抱多模態大模型。
* **動作** ：拔除 DeepFace、OpenCV、TensorFlow 等所有笨重套件。將架構改為「直接讓 LLM 看圖辨識心情」，後端正式從「本地推論中心」轉為「多模態請求整理與代理層」。
* **因果轉折** ：架構輕量化後，系統具備了處理更複雜輸入的能力，加上單純的「情緒測試」缺乏商業價值，促使我們在 Phase 5 引入具體的應用情境。

### Phase 5：轉型「AI 保險員」與行動端適配

* **目標** ：包裝商業情境，並支援語音對話與手機端測試。
* **動作** ：加入麥克風收音機制。因為手機測試碰到 HTTPS / MediaDevices 限制，導致功能直接卡死，所以開始調整 Uvicorn 綁定 IP，並引入 Chrome Flags 白名單與 Ngrok 內網測試策略。
* **因果轉折** ：隨著語音、鏡頭、狀態提示等 UI 元素不斷增加，前端 DOM 開始頻繁被改壞，引發了最後的 UX 收斂與重構。

### Phase 6：UX 深度優化與功能擴充

* **目標** ：解決介面混亂與互動生硬的問題，讓 Demo 更像完整產品。
* **動作** ：實作 PDF 保單上傳解析、優化對話紀錄為可滑動的持久化歷史、將鏡頭改為可拖曳浮動視窗，並嘗試導入 SSE (Server-Sent Events) 串流機制降低延遲。

## 4. 遇到的關鍵問題與踩坑紀錄

這部分整理為可快速檢索的 Debug 索引，記錄了開發過程中的重大卡關點：

* **Numpy Float32 序列化報錯**
  * **類型** ：序列化 / API 格式
  * **影響範圍** ：回傳失敗 (HTTP 500)
  * **是否已根治** ：是
  * **細節** ：DeepFace 回傳的信心分數是 numpy 型態，FastAPI 內建的 JSON Encoder 無法解析。解法是在回傳前用 `float()` 強制轉型為 Python 原生型態（雖然後期隨架構轉換已移除該模組）。
* **DeepFace 判斷「笑」為「驚訝」**
  * **類型** ：模型品質
  * **影響範圍** ：辨識錯誤 / UX 邏輯錯亂
  * **是否已根治** ：是（透過重構捨棄）
  * **細節** ：FER-2013 訓練資料集的局限性，導致張嘴輪廓極易被誤判為 `surprise` 或極端的 `angry`，對亞洲人臉適應不良。最終果斷放棄本機辨識。
* **手機瀏覽器封鎖 MediaDevices**
  * **類型** ：瀏覽器權限與安全性
  * **影響範圍** ：行動端完全不可用（觸發 `NotSupportedError`）
  * **是否已根治** ：部分（需環境配合）
  * **細節** ：現代瀏覽器安全規範限制 `getUserMedia` 只能在 `https://` 或 `localhost` 執行。目前需透過 `chrome://flags/#unsafely-treat-insecure-origin-as-secure` 加白名單，或透過 `ngrok` 提供 HTTPS 隧道來進行內網測試。
* **DOM 元素遺失導致流程卡死**
  * **類型** ：前端 DOM 狀態管理
  * **影響範圍** ：流程中斷（卡在「思考中」）
  * **是否已根治** ：是
  * **細節** ：Vibe coding 過程頻繁修改 HTML UI，移除了 `#ai-subtitles` 等 ID，但 JS 仍試圖操作它們，導致 Uncaught TypeError 終止非同步函數。解法是全面增加元素存在與否的防呆檢查 `if(element) {...}`。

## 5. 為什麼後來改架構

專案最核心的轉折點： **從「本機端組合式 AI」轉向「雲端多模態原生 AI」** 。

原本方案的致命傷在於 **準確度太差** 與  **環境太重** 。DeepFace 依賴傳統模型泛化能力不足，即使套用大量 Hack 依然無效；且依賴 TensorFlow 導致安裝極慢、易發版本衝突。

隨著 Gemini 2.5 Flash 證明了其多模態能力，我們果斷刪除所有本機 ML 套件。後端主要職責從「自己做情緒推論」改成「接收媒體資料、整理 Prompt、呼叫 Gemini、回傳標準格式結果」。

### 5.1 舊方案與新方案對照

| 面向                 | 舊方案 (Phase 1-3)             | 新方案 (Phase 4-6)           |
| -------------------- | ------------------------------ | ---------------------------- |
| **情緒判斷**   | DeepFace + FER-2013            | Gemini 多模態 Vision         |
| **語音處理**   | 無或極少                       | MediaRecorder + Gemini STT   |
| **後端角色**   | 模型推論執行者                 | 多模態請求整理與代理層       |
| **核心依賴**   | TensorFlow / OpenCV / DeepFace | FastAPI + Gemini SDK         |
| **部署難度**   | 高（需編譯 C++ 庫與大記憶體）  | 極低（純 Python 輕量服務）   |
| **實際準確度** | 不穩定、極易誤判               | 明顯較佳，具備上下文綜合能力 |

## 6. 最終版本的核心架構

### 6.1 目前的核心技術決策

* **保留 Vanilla JS** ：前期迭代速度快，方便直接操作 DOM 與 Media API（但也帶來了後期狀態管理的挑戰）。
* **採用 FastAPI** ：因應快速處理 `multipart/form-data`、CORS 設定與輕量 API 封裝的需求。
* **放棄本機情緒模型** ：實測準確度與維運（安裝/編譯）成本皆不合格。
* **以 Gemini 為多模態中樞** ：它能同時覆蓋語音、圖像、文件與對話生成，極大地降低了系統後端的複雜度。

### 6.2 架構模組劃分

* **前端** ：前端目前以 `FormData` 為主，整合語音、截圖與文件等輸入資料送至後端；實際夾帶內容會依當前功能開關與頁面狀態而有所差異。管理拖曳式 PDF 上傳與浮動鏡頭 UI。
* **後端** ：轉為輕量 API 代理層。接收請求後，將資料轉化為 Gemini SDK 接受的格式，注入「AI 保險員」人設系統 Prompt，並代理外部 API 請求。
* **模型服務** ：目前版本以 Gemini 2.5 Flash 為核心，負責多模態理解（STT、Vision、Document Reading）與最終回覆的生成。

## 7. 最終資料流 (Data Flow)

這是一次完整對話互動的資料流動過程：

1. **觸發聆聽** ：使用者點擊開始，`MediaRecorder` 開始錄製語音。
2. **畫面擷取** ：在說話期間，前端定時透過 Canvas 截圖，暫存壓縮過的 JPEG 照片。
3. **資料打包發送** ：使用者講完 -> 停止錄音 -> 前端將語音檔、照片與當前上下文（及 PDF）打包進 `FormData` -> POST 送至後端。
4. **後端重組請求** ：FastAPI 收到資料，轉化為 Gemini 多模態陣列，並組合 Prompt。
5. **雲端推論** ：Gemini 同步聽音檔、看照片、查閱文件，產出回覆。
6. **回傳與解析** ：後端會將 Gemini 回傳結果整理成前端可消化的標準格式；在不同階段中，**曾使用一般 JSON 回傳，也曾朝更即時的串流輸出（SSE / chunked）方式演進**以降低延遲。
7. **UI 更新與輪迴** ：前端顯示回覆、切換情緒徽章。系統已朝「自動下一輪互動」設計，但實際行為依當前前端狀態與使用者操作控制而定。

## 8. 這個專案真正學到什麼

* **「組合式 AI」不如「多模態大模型」** ：傳統專用小模型在未經微調的現實場景中極度脆弱。直接把原始多模態資料（音/影/圖）餵給大模型，綜合理解力遠勝拼湊架構。
* **Vibe Coding 必須搭配嚴謹的狀態管理** ：因為 UI 開始複雜化、DOM 常被改壞，所以後續極需更明確的狀態管理機制，否則極易引發連鎖報錯。
* **瀏覽器安全策略不可輕忽** ：HTTPS / Localhost 限制是 Web Media 應用的硬傷，提早確立 Ngrok 穿透或內網測試規範，能省下大量 Debug 時間。
* **重構的勇氣** ：專案從難用到順暢的關鍵，在於「勇敢刪除程式碼」。把整個後端 ML 環境連根拔起，反而造就了最穩定、擴展性最強的版本。

## 9. 後續可以怎麼優化

1. **導入 WebRTC 或 WebSocket 串流對話**
   * *目前限制* ：半雙工模式，延遲感仍存在。
   * *優化方向* ：改用 Gemini Live API (WebRTC) 建立即時雙向串流。
2. **前端框架重構 (React/Vue)**
   * *目前限制* ：Vanilla JS 狀態疊加後難以維護。
   * *優化方向* ：將 UI 元件化，利用框架的狀態管理來控制渲染。
3. **實作 RAG 知識庫**
   * *目前限制* ：每次都要重新上傳或讓 LLM 重讀整個 PDF，消耗 Token 且效率低。
   * *優化方向* ：導入向量資料庫預先建立保單知識庫，先檢索片段再餵給 LLM。
4. **語音打斷機制 (Barge-in)**
   * *目前限制* ：AI 語音回覆過長時，使用者無法自然插話。
   * *優化方向* ：前端播放語音時持續監聽麥克風，偵測到人聲大於特定閾值即截斷播放。

## 10. 目前系統邊界與定案狀態

為了利於後續接手與開發，以下明確定義當前系統的能力邊界：

### ✅ 目前已可穩定展示的能力（已定案）

* 基本語音輸入與錄音上傳流程。
* 鏡頭畫面擷取與多張截圖整合分析。
* 以 Gemini 2.5 Flash 為核心的多模態理解與回覆生成。
* 基礎 PDF 保單上傳與單次情境式問答。
* 前端純 HTML/CSS/Vanilla JS 實作，後端 FastAPI 輕量封裝。
* 透過 `FormData` 傳送實體檔案，可操作的介面與部分行動端（內網）測試流程。

### 🚧 目前仍屬實驗性或尚未完全定案的能力

* **真正低延遲的即時雙向串流對話** （目前雖有嘗試 SSE 串流，但離真實電話體驗仍有距離）。
* **穩定可用的 Barge-in 語音打斷** （目前尚無法完美中斷 AI TTS 播報）。
* **完整長期記憶與多輪上下文管理** （歷史對話整合機制尚待優化）。
* **正式 RAG 知識庫取代單次 PDF 解析** 。
* **前端框架化與完整狀態管理重構** （尚未脫離 Vanilla JS 泥淖）。
