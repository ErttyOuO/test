# 客服與保單上傳資料庫功能移植安全草案

## 目標

本次預計從 `客服與保單上傳資料庫/` 擷取以下三塊「較新版本」功能，整理後再移植到目前工作目標資料夾：

1. 保單目錄
2. 智慧保單
3. 客服功能優化

本草案的重點不是直接覆蓋，而是先建立一份安全的中繼副本，避免原始檔與目標檔在比對、調整、回填過程中被誤改或毀損。

## 建議原則

建議採用「空資料夾骨架 + 分批複製相關檔案」方式，不要直接整包複製來源資料夾，也不要直接覆蓋 `重新設計版面樣式/` 現有檔案。

原因如下：

1. 兩邊目前都已存在大量同名檔案，如 `dm.html`、`policy.html`、`script.js`、`style.css`、`auth.js`、`server.js`。
2. 這些檔案多半不是單一功能專用，而是共用樣式或共用腳本，直接覆蓋很容易把近期首頁、輪播、主題切換等既有修正一起蓋掉。
3. `客服與保單上傳資料庫/` 與 `重新設計版面樣式/` 的資料路徑並不完全相同，例如保單資料來源就有 `保單搜尋/policy_data.json` 與根目錄 `policy_data.json` 的差異，直接搬移容易造成路徑錯誤。

## 建議作法

### 第一階段：先建立空的中繼子資料夾

建議在 `重新設計版面樣式/` 底下先建立一個完全獨立的中繼資料夾，例如：

`重新設計版面樣式/功能移植暫存_客服與保單上傳資料庫/`

先只建立空骨架，不先放任何來源檔：

```text
功能移植暫存_客服與保單上傳資料庫/
  01_保單目錄/
  02_智慧保單/
  03_客服功能/
  04_共用資源/
  05_比對紀錄/
```

這樣做的目的：

1. 原始來源不動。
2. 正式目標資料夾不直接受影響。
3. 每一類功能可以分開比對、分開驗收、分開回填。

### 第二階段：只複製功能核心檔，不直接複製整包

先把來源中與功能直接相關的檔案，複製進上述中繼資料夾，再做 diff 與整合。

## 建議優先複製的檔案

### A. 保單目錄

來源核心檔案建議先複製：

1. `客服與保單上傳資料庫/dm.html`
2. `客服與保單上傳資料庫/dm_policy.js`
3. `客服與保單上傳資料庫/保單搜尋/policy_data.json`

可能連動、需要一併比對的共用檔：

1. `客服與保單上傳資料庫/script.js`
2. `客服與保單上傳資料庫/style.css`
3. `客服與保單上傳資料庫/auth.js`

注意事項：

1. 目前 `重新設計版面樣式/dm_policy.js` 讀的是根目錄 `policy_data.json`。
2. 來源版 `客服與保單上傳資料庫/dm_policy.js` 讀的是 `保單搜尋/policy_data.json`。
3. 這表示保單目錄功能即使版面相同，資料路徑仍有結構差異，不能直接覆蓋。

### B. 智慧保單

來源核心檔案建議先複製：

1. `客服與保單上傳資料庫/policy.html`

可能連動、需要一併比對的檔案：

1. `客服與保單上傳資料庫/script.js`
2. `客服與保單上傳資料庫/style.css`
3. `客服與保單上傳資料庫/auth.js`
4. `客服與保單上傳資料庫/idb-files.js`
5. 若頁面引用保單分析資料或上傳流程，再視實際引用補複 `app.py`、`Function/`、`server/` 內相關檔案

注意事項：

1. `policy.html` 很可能依賴共用的版型、登入檢查、按鈕互動與 FAQ 浮動按鈕。
2. 若只搬 `policy.html` 不搬相依腳本，畫面可能能開，但互動功能會殘缺。

### C. 客服功能優化

目前已確認的客服相關結構，建議先複製以下來源檔案作比對：

1. `客服與保單上傳資料庫/script.js`
2. `客服與保單上傳資料庫/style.css`
3. `客服與保單上傳資料庫/dm.html`
4. `客服與保單上傳資料庫/budget.html`

補充說明：

1. `重新設計版面樣式/` 目前已存在 FAQ / 客服彈窗結構與對應函式。
2. 客服功能優化不一定是單一頁面更新，可能散落在頁面 HTML、共用 JS、共用 CSS。
3. 因此這一塊應該以「功能片段合併」為主，而不是整個檔案直接覆寫。

## 建議移植流程

### 步驟 1：建立中繼空骨架

先建立：

`重新設計版面樣式/功能移植暫存_客服與保單上傳資料庫/`

只建立子資料夾，不複製任何正式檔到目標頁面。

### 步驟 2：分功能複製來源檔到中繼區

建議複製規則：

1. 保單目錄相關檔放進 `01_保單目錄/`
2. 智慧保單相關檔放進 `02_智慧保單/`
3. 客服優化相關檔放進 `03_客服功能/`
4. 共用 `script.js`、`style.css`、`auth.js` 這類高風險檔放進 `04_共用資源/`
5. 將比對結論與風險註記寫到 `05_比對紀錄/`

### 步驟 3：先做差異比對，再決定回填方式

每一類功能都先做以下判斷：

1. 哪些是可以整頁替換的獨立頁面。
2. 哪些只是片段更新，必須手動摘出功能區塊。
3. 哪些共用檔有高度耦合，只能局部合併，不能覆蓋。

### 步驟 4：採「片段回填」而非「整檔覆蓋」

回填優先順序建議如下：

1. 先回填保單目錄頁面的純功能區塊。
2. 再回填智慧保單頁面的新版區塊。
3. 最後才處理 `script.js`、`style.css` 這種共用檔，避免連帶破壞首頁與其他頁面。

### 步驟 5：複製完成後，處理 `重新設計版面樣式/` 原有相關檔案

你提到複製完成後，要把 `重新設計版面樣式/` 原本已有的相關檔案刪除，避免重複。這個方向是對的，但我不建議第一時間直接永久刪除，建議改成兩段式處理：

1. 先把舊檔移到隔離區。
2. 驗證新版本沒問題後，再做正式刪除。

建議先建立一個舊檔隔離資料夾，例如：

`重新設計版面樣式/_舊版待刪除_功能移植/`

可再依功能分流：

```text
_舊版待刪除_功能移植/
  01_保單目錄/
  02_智慧保單/
  03_客服功能/
  04_共用資源/
```

這樣做的原因：

1. 可以避免正式目錄裡同時出現新舊兩套相關檔案。
2. 若移植後發現功能缺漏，可以立即回補，不必重新找舊版。
3. 比直接刪除安全很多，尤其是 `script.js`、`style.css` 這種共用檔。

## 複製後要處理的「原有相關檔案」原則

### 可以從正式目錄移出或淘汰的對象

如果新版本已完成回填並驗證通過，以下這些「被新版取代的舊檔或舊片段來源」就可以從 `重新設計版面樣式/` 正式工作區移出：

1. 舊版 `dm.html`
2. 舊版 `policy.html`
3. 舊版 `dm_policy.js`
4. 若已被新版完整接手的客服 FAQ 區塊來源頁
5. 若已拆出並重新整合完成的暫存複製檔

但要注意：

1. `script.js` 不應該因為有新版片段就整份刪除，除非你已經完成完整重構與驗證。
2. `style.css` 不應該因為搬了客服樣式就整份刪除，應該只清掉重複區塊或已淘汰區塊。
3. `auth.js` 同理，不建議直接刪整份，只能在確認新舊邏輯重疊後移除多餘片段。

## 建議的刪除策略

### A. 可直接替換型檔案

這類檔案如果確定新版已完整覆蓋功能，可以採：

1. 先把舊檔移到 `_舊版待刪除_功能移植/`
2. 在正式目錄放入新版檔
3. 開啟頁面驗證功能、連結、按鈕與資料讀取
4. 驗證無誤後，再刪掉隔離區中的舊檔

較可能適用的檔案：

1. `dm.html`
2. `policy.html`
3. `dm_policy.js`

### B. 共用整合型檔案

這類檔案不建議直接刪除原檔，而是應該：

1. 保留現行正式檔作主檔
2. 從來源版抽出需要的新功能片段
3. 合併進現有正式檔
4. 最後只刪除重複片段、廢棄 class、廢棄函式、未再引用的資源

較可能適用的檔案：

1. `script.js`
2. `style.css`
3. `auth.js`

### C. 暫存與比對用副本

這一類在功能正式上線後，應該優先清理，避免你之後看到重複檔名混淆：

1. `功能移植暫存_客服與保單上傳資料庫/` 內的純複製副本
2. 測試用比對檔
3. 已確認不再使用的中間版 html / js / css

## 我會建議的實際順序

如果你要避免重複，我建議順序改成這樣：

1. 先建立空的中繼資料夾。
2. 把來源功能檔複製到中繼區。
3. 將新版功能合併到 `重新設計版面樣式/` 正式檔。
4. 把 `重新設計版面樣式/` 中被取代的舊檔先移到 `_舊版待刪除_功能移植/`。
5. 驗證頁面正常後，再正式刪除隔離區中的舊檔與暫存副本。

這樣你仍然能達成「避免重複」的目的，但不會在還沒驗證前就把原本可回復的檔案清掉。

## 風險清單

### 高風險檔案

1. `script.js`
2. `style.css`
3. `auth.js`

這三類檔案通常影響整站，不建議直接用來源版整份覆蓋。

### 中風險檔案

1. `dm.html`
2. `policy.html`

雖然是單頁，但仍可能引用共用函式、共用樣式與共用浮動元件。

### 資料風險

1. `policy_data.json` 的放置位置與引用路徑可能不同。
2. 若未同步調整 JS 讀取路徑，頁面會出現載入失敗或資料空白。

## 我建議的執行策略

建議不要直接從 `客服與保單上傳資料庫/` 把整包功能一次覆蓋到 `重新設計版面樣式/`。

較安全的做法是：

1. 先在 `重新設計版面樣式/` 建一個空的中繼子資料夾。
2. 把保單目錄、智慧保單、客服功能相關檔案分流複製進去。
3. 逐項比對後，只把確定較新的功能片段合併回正式檔。
4. 將 `重新設計版面樣式/` 內被取代的原有相關檔先移到舊檔隔離區，不要第一時間永久刪除。
5. 每完成一塊就先做一次頁面驗證，再進下一塊。
6. 驗證通過後，再刪除舊檔隔離區與功能移植暫存區內不再需要的檔案。

## 建議下一步

如果你同意這個流程，下一輪我可以直接幫你做兩件事：

1. 先建立中繼空骨架資料夾。
2. 再建立舊檔隔離資料夾，讓你之後可以把 `重新設計版面樣式/` 原有相關檔案移出而不是硬刪。
3. 接著依照「保單目錄 / 智慧保單 / 客服功能」三類，把建議複製清單細化成可直接執行的名單。
