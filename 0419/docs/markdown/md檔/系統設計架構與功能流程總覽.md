# 系統設計架構與功能流程總覽

最後更新：2026-03-27
文件用途：
- 提供新功能開發的共同參考基準
- 說明目前系統範圍、設計架構、資料流程與功能目的
- 降低新成員接手成本，避免重複開發與流程中斷

---

## 1. 專案範圍與定位

IAFM 是保險與財務管理整合專案，包含多個階段版本。
目前建議以 重新設計版面樣式 作為主線開發版本；原先 AI 擬人視訊相關能力已併入主線，其它資料夾多為歷史版本、功能實驗或移植樣板。

### 1.1 工作區範圍（高層）
- 主線版本：重新設計版面樣式
- 歷史版本與分支：保單詢問按鈕、保單檔案類型修正、頁面與功能改善、舊版本 等
- 其它輔助 / 實驗專案：OCR+LLM語言模型爬蟲 等

### 1.2 主線版本目標
- 建立 AI 保險決策平台首頁與核心功能入口
- 提供保單整理、保單比較、社群互動、預算規劃、會員與任務機制
- 整合 AI 對話、語音、情緒偵測與保單摘要能力

---

## 2. 系統架構總覽

### 2.1 架構分層
1. 前端展示層
- 位置：重新設計版面樣式/frontend/pages、frontend/styles、frontend/scripts
- 內容：首頁、保單頁、社群頁、預算頁、會員頁、任務頁、後台頁等
- 角色：提供使用者操作介面與視覺互動

2. 前端共用邏輯層
- 位置：重新設計版面樣式/frontend/scripts/script.js、auth.js
- 內容：全站導覽、登入註冊、頁面初始化、社群與保單互動、首頁行為
- 角色：整合各頁行為，降低每頁重複程式

3. API 與資料服務層（Node）
- 位置：重新設計版面樣式/backend/services/node-server
- 內容：Express API、MongoDB 連線、資料模型
- 角色：提供保單、貼文、組合建議、客服等資料操作

4. AI 服務層（Face Backend）
- 位置：重新設計版面樣式/backend/services/face_backend
- 內容：FastAPI、語音轉文字、情緒分析、Gemini 對話串流、保單摘要
- 角色：承接多模態 AI 能力

5. 補充 AI 服務層（Unified Chat）
- 位置：重新設計版面樣式/backend/services/unified-chat
- 內容：整合式聊天後端腳本
- 角色：支援聊天組件整合與擴充

### 2.2 主要資料儲存
- MongoDB：使用者、貼文、保單、組合建議等結構化資料
- LocalStorage：前端登入狀態、部分使用者偏好與展示用快取
- 記憶體暫存：Face Backend 的保單摘要暫存（單機執行時）

## 3. 目錄結構（主線）

- 重新設計版面樣式/index.html：主頁入口
- 重新設計版面樣式/frontend/pages：功能頁與後台頁
- 重新設計版面樣式/frontend/styles/style.css：全站主樣式
- 重新設計版面樣式/frontend/scripts/script.js：全站功能初始化與頁面邏輯
- 重新設計版面樣式/frontend/scripts/auth.js：帳號與權限核心
- 重新設計版面樣式/backend/services/node-server/server.js：Node API 主服務
- 重新設計版面樣式/backend/services/node-server/models：資料模型
- 重新設計版面樣式/backend/services/face_backend/main.py：AI 對話與情緒後端
- 重新設計版面樣式/docs/markdown/md檔：系統文件與整合說明

## 4. 系統流程（一步一步）

## 4.1 使用者主流程（產品視角）
1. 進入主頁
- 看見 AI 保險決策平台定位
- 從 Hero 與功能卡選擇入口（例如智慧保單）

2. 登入或註冊
- 透過快速入口完成身份驗證
- 系統建立或讀取使用者資料

3. 選擇任務情境
- 保單比較：看方案方向與差異
- 智慧保單：上傳保單並取得摘要
- 會員功能：解鎖進階能力
- 每日任務：建立長期互動與回訪

4. 啟動 AI 分析
- 可於諮詢頁用文字或語音互動
- 系統整合情緒線索、歷史對話、保單摘要進行回應

5. 形成決策與持續優化
- 取得分析結果與下一步建議
- 用社群、預算、會員工具持續迭代保障策略

## 4.2 技術流程（請求視角）
1. 前端頁面載入
- script.js 依頁面路徑啟動對應初始化函式

2. 認證與權限處理
- auth.js 管理登入狀態與角色
- Header 與快捷入口同步狀態

3. 業務資料讀寫
- 前端呼叫 Node API
- Node API 經由 Mongoose 寫入或讀取 MongoDB

4. AI 能力呼叫
- 前端將音訊、影像、文字送到 Face Backend
- Face Backend 呼叫 Gemini 相關服務並串流回傳

5. 前端渲染結果
- 即時更新回覆、情緒顯示、保單分析內容與互動狀態

## 5. 現有功能、目的、使用方式與可能成效

### 5.1 主頁與導流
- 功能目的：清楚傳達平台價值，讓使用者快速進入對應功能
- 如何使用：點選 Hero 按鈕與功能卡進入對應頁面
- 可能成效：降低首次使用者迷航，提升轉換率

### 5.2 智慧保單
- 功能目的：把保單內容轉為可理解的重點
- 如何使用：上傳保單，查看摘要與細節卡
- 可能成效：縮短理解條款時間，提升保障缺口辨識效率

### 5.3 AI 諮詢與情緒偵測
- 功能目的：提供更貼近情境的對話建議
- 如何使用：文字或語音提問，可搭配影像情緒分析
- 可能成效：提升互動品質與建議可採納性

### 5.4 保單比較與目錄
- 功能目的：快速比較方案與篩選方向
- 如何使用：依類別與條件查看保單與差異
- 可能成效：減少決策時間，提升方案匹配度

### 5.5 社群互動
- 功能目的：收集真實經驗，降低資訊落差
- 如何使用：瀏覽貼文、收藏、互動、篩選
- 可能成效：建立信任感，提升黏著與回訪

### 5.6 預算規劃
- 功能目的：將保障需求轉為可執行預算
- 如何使用：輸入條件，查看推薦預算分配
- 可能成效：降低超支或保障不足的風險

### 5.7 會員與任務機制
- 功能目的：建立留存與分層服務
- 如何使用：升級會員、參與每日任務與獎勵循環
- 可能成效：提高留存率與長期使用價值

## 6. 目前系統設計原則

1. 可維護
- 以共用腳本集中全站初始化邏輯
- 以服務分層切分前端、資料服務、AI 服務

2. 可擴充
- 前端可新增頁面並透過路徑初始化掛載
- 後端可在 node-server 新增模型與 API
- AI 服務可擴充情緒、語音、保單分析能力

3. 可驗證
- 每個主要功能均有可觀察的輸入、處理、輸出
- 可由日誌與 API 回應追蹤流程狀態

---

## 7. 新功能開發參考流程（建議遵循）

1. 先定義需求
- 目標使用者
- 要解決的問題
- 成功指標（例如完成率、回訪率、分析準確感）

2. 定義模組位置
- 前端頁面在 frontend/pages
- 前端邏輯在 frontend/scripts
- API 在 backend/services/node-server
- AI 行為在 backend/services/face_backend

3. 建立資料契約
- 請先定義輸入欄位、回傳格式、錯誤訊息
- 前後端同步確認資料結構再開發

4. 實作與整合
- 先完成 API，再接前端互動
- 使用小步驟提交，避免一次改動過大

5. 驗證與文件更新
- 檢查主要流程與例外流程
- 更新本文件與對應模組說明

---

## 8. 開發注意事項

1. 主線優先
- 新功能請優先落在 重新設計版面樣式，避免分支版本再次分裂

2. 資料一致性
- 同一功能避免同時存在多套入口與多套資料來源

3. 效能與穩定
- 前端避免過多同步阻塞
- AI 服務注意外部 API 延遲與失敗回退

4. 安全與配置
- 機密資訊應改為環境變數管理
- 不在前端暴露敏感金鑰

---

## 9. 後續可優先改善項目

1. 保單資料流統一
- 打通主頁、智慧保單、AI 諮詢的同一資料來源

2. 模組拆分
- script.js 可逐步拆為多個功能模組，降低單檔複雜度

3. 文檔標準化
- 每個核心模組建立固定格式文件：目的、輸入、輸出、依賴、風險

4. 觀測指標
- 補上功能使用率、轉換率、回訪率等追蹤指標
