更新日期：2026-04-05

---

## 1. 專案定位與現況

本資料夾可視為 IAFM 專案目前主線版本，定位為「AI 保險決策平台」。

目前系統不是純靜態頁面，實際上由以下層次組成：

1. 前端頁面層（frontend/pages）
2. 前端共用邏輯（frontend/scripts）
3. Node.js 服務層（backend/services/node-server）
4. Flask AI 對話層（backend/services/flask-app）
5. FastAPI 面容與語音情緒層（backend/services/face_backend）
6. Unified Chat 補充服務層（backend/services/unified-chat）

---

## 2. 目錄責任總覽

- index.html
  - 專案入口頁，base 指向 frontend/pages
  - 首頁含 Hero、決策流程、旅程、功能卡

- frontend/pages
  - 使用者前台與管理後台頁面集合

- frontend/scripts
  - script.js：全站互動與各頁初始化
  - auth.js：登入、角色、使用者資料與權限控制
  - ad-sdk.js / idb-files.js：廣告與素材儲存相關

- frontend/components/unified-chat-components
  - AI 對話整合元件（情緒偵測、語音、Avatar、對話管理）

- backend/services/node-server
  - 主要 Web/API 伺服器，連 MongoDB，並可自動帶起 Flask

- backend/services/flask-app
  - AI 文字對話後端（Vertex 為主、Gemini API Key 為備援）

- backend/services/face_backend

- backend/services/unified-chat

  - Windows 啟動腳本（start_all.ps1 等）

- docs/markdown、docs/md
  - 技術文件、改版紀錄、整合說明與本整理檔

## 7. 主要資料流程（功能觀點）

### 7.1 AI 諮詢流程（文字）

1. 前端送出訊息到 Flask /chat
2. Flask 優先走 Vertex，失敗時 fallback 到 Gemini API Key
3. 回傳純文字建議

### 7.2 保單上傳與摘要流程

1. 前端上傳保單（policy 頁）
2. Node 寫入 MongoDB（Policy）
3. 前端或 FastAPI 進一步讀取摘要與 AI 分析
4. 需要時回寫 aiReply 快取到資料庫

### 7.3 情緒感知對話流程

1. 前端送 audio/images 到 FastAPI /chat
2. FastAPI 進行 STT、情緒分析、候選保單排序
3. 以 SSE 串流回傳 init/chunk/done 給前端

## 11. 首頁 UI 設計與配色補充（本次新增）

### 11.1 設計來源（實作檔）

- index.html（首頁 HTML 結構，`<body class="home-saas">`）
- frontend/styles/style.css（`body.home-saas` 視覺系統）
- frontend/scripts/script.js（首頁初始化：`attemptAutoConnect`、`initCompanyAutocomplete`、`initHomeAdCarousel`）

### 11.2 首頁資訊架構（IA）

首頁主體在 `main.saas-main`，目前由四個區塊構成：

1. saas-hero：主標、主 CTA、AI 顧問視覺與對話預覽
2. ai-decision：輸入層 → 分析中 → 建議卡的決策流程
3. journey：三步驟用戶旅程（發現問題／AI 分析／做出決策）
4. features-late：補充功能卡（保單比較、智慧保單、會員、每日任務）

### 11.3 首頁配色 Token（home-saas）

目前首頁主要色票可視為「藍青科技感」系統：

- `--saas-primary`: #3B82F6
- `--saas-accent`: #56F3FF
- `--saas-bg`: #f4f8ff
- `--saas-surface`: #ffffff
- `--saas-text`: #0f172a
- `--saas-muted`: #5b667a
- `--saas-dark`: #071428
- `--saas-dark-2`: #0f223d

搭配策略：

1. 亮底閱讀區（hero / journey / features）提升資訊可讀性
2. 深底決策區（ai-decision）做節奏反差，凸顯「AI 引擎」段落
3. 主按鈕使用藍→青漸層，次按鈕使用淺底描邊

### 11.4 動效與視覺節奏

- 使用 `[data-reveal]` + `.is-visible` 作為區塊進場
- `feature-visual` 可進入 `float-active` 漂浮狀態
- 已具備 `prefers-reduced-motion` 與 `.reduced-motion` 降動效處理

### 11.5 重要註記：目前僅首頁做視覺改版

本輪樣式重構的「設計語言更新」主要鎖定首頁（`index.html` + `home-saas` 區塊）。

其他頁面（如 `consulting.html`、`policy.html`、`community.html` 等）目前仍沿用既有結構與樣式體系，尚未進入同等級的全站視覺翻新。

## 13. 黑白樣式錯置原因與移除策略（為下階段改版準備）

### 13.1 錯置來源（目前程式狀態）

現有樣式同時存在：

1. `home-saas`（首頁新視覺）
2. `home-future`（舊未來感深色主題）
3. `body:not(.home-future)`（舊亮底分支）
4. `data-theme="dark"`（全域深色覆蓋）

且 `script.js` 的 `initThemeToggle()` 在未指定固定主題時，預設 `app-theme = home-future`，會對 `body` 加上 `home-future` 與 `data-theme="dark"`。這是目前「看起來像黑白/深色樣式互相蓋來蓋去」的核心原因之一。

### 13.2 建議移除路徑（文件先行，程式可下一步執行）

1. 首頁固定主題
- 讓首頁只吃 `home-saas`（停用首頁上的 `home-future` 注入）

2. 主題切換範圍縮限
- Theme Toggle 先限定於非首頁頁面，避免覆蓋首頁改版成果

3. 樣式分層清理
- 將 `home-future`、`not(.home-future)` 樣式標記為 legacy 區
- 分批搬移到獨立檔，逐頁替換後再移除

4. 驗收基準
- 首頁在清空 localStorage 後仍維持 `home-saas` 外觀
- 切換帳號/重整頁面不再把首頁帶回舊深色主題
- 既有功能（登入、導航、搜尋、諮詢入口）行為不變
