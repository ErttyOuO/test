## 三、尚未整合項目（發現的問題）

---

### 3-1. ❌ `script.js` — 缺少 10 個 AI 保單分析函式

來源版有以下函式，**目標版完全沒有**：

| 函式名稱 | 用途 | 影響頁面 |
|---------|------|---------|
| `async function loadPolicies()` | 從 API（`localhost:3000/api/policies`）**真實載入**使用者保單 | `policy.html` |
| `async function fetchAIPolicySummary(policy)` | 呼叫 AI 生成保單條款摘要並顯示在 `.term-body` 區塊 | `policy.html` |
| `async function callGeminiAI(prompt, fileData, fileType, apiKey, modelName)` | 底層 Gemini API 呼叫（直接傳 API Key） | `policy.html` |
| `async function safeCallGeminiAI(prompt, fileData, fileType)` | 安全封裝版 Gemini 呼叫（自動重試、錯誤處理） | `policy.html` |
| `function calculateSimpleHash(str)` | 計算檔案雜湊用於快取 AI 結果 | `policy.html` |
| `function simulateAITextAnalysis(text, fileName, fileHash)` | 本地模擬 AI 文字分析（離線 fallback） | `policy.html` |
| `async function extractTextFromPDF(dataUrl)` | 使用 PDF.js 從 PDF 提取文字供 AI 分析 | `policy.html` |
| `function processUpload(file)` | **完整的 AI 驅動保單上傳流程**（含訪客限制、進度顯示、OCR/PDF 分析） | `policy.html` |
| `async function simulateAIResponse(text)` | AI 客服回應模擬（目標版有同名函式但邏輯不同） | `budget.html`、`consulting.html` |
| `async function checkLocalImageServices()` | 啟動時檢查本地圖片生成服務是否可用 | 全域 |

#### 重要差異：`processUpload` 函式

| | 來源版 | 目標版 |
|--|--------|--------|
| 參數 | `file`（File 物件） | `fileName`（純字串） |
| 邏輯 | 訪客使用限制 → FileReader → PDF/圖片分支處理 → AI 分析 → 儲存至 API | 只顯示 UI 假進度條，無實際分析 |
| 功能完整度 | **完整** | **空殼** |

#### 重要差異：`initPolicyFeatures` 重複定義

- 目標版 `script.js` 在 **行 1851** 和 **行 3228** 均有 `initPolicyFeatures` 定義
- 兩個版本都使用**假資料（硬編碼 4 筆保單）**
- 來源版的 `initPolicyFeatures` 會呼叫 `loadPolicies()` 從 API 取得**真實資料**
- **問題**：重複定義導致後面的函式覆蓋前面；且兩者都缺少真實 API 載入邏輯

## 六、根本原因分析

上次整合（Phase 5）主要聚焦在 **HTML 結構補丁**（policy.html 的 FAQ/PDF.js、budget.html 的 FAQ modal），但遺漏了三個核心層面：

1. **JavaScript 業務邏輯層**：AI 保單分析的完整上傳→分析→顯示流程（`processUpload`、`extractTextFromPDF`、`safeCallGeminiAI`、`fetchAIPolicySummary`）未整合，導致保單頁只有 UI 但功能是假資料。
2. **CSS 樣式層**：AI 分析結果的顯示樣式（`.ai-analysis-output` 系列）缺失，即使整合了 JS 邏輯，分析結果也會無格式顯示。
3. **獨立模組**：`policy_tab.js` 作為獨立 IIFE 完全被遺漏，保單目錄的搜尋/篩選互動功能無法運作。
